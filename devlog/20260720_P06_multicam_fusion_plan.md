# P06: 멀티 카메라 융합 — 동기화·정합·개체 매칭 (PitchWatch)

- 날짜: 2026-07-20
- 종류: 계획
- 관련: [P03](20260719_P03_highlights_roadmap_plan.md) (검출·이벤트 기반),
  [P05](20260720_P05_app_split_plan.md) (PitchWatch 소속 기능)

## 배경 (데이터 현황)

- 대부분의 경기에 **두 소스가 이미 존재**: (a) GoPro 듀얼 리그 파노라마
  (중앙선 근처 사이드라인), (b) **Sony AX700** — 반대편 사이드라인,
  경기장 전체를 한 번에 못 보지만 **화질 우수**.
- 상호보완이 정확히 맞물림: 파노라마의 원경 절반 = AX700 의 근측.
  원경 공 검출 공백·원측 선수 화질·등번호 OCR 커버리지가 반대편
  시점으로 메워진다.

## 핵심 설계

### 동기화 — 2단계 (호각만으로는 부족)

1. **거친 동기화 + 클럭 드리프트**: 호각 트랙(.whistle.json) 상호상관.
   음속(343m/s) 때문에 심판 위치에 따라 호각별 오프셋이 ±0.3s 흔들림
   → 여러 호각의 오프셋에 **선형 시계 모델 (a + b·t)** 피팅 (두 기기
   클럭은 수 ppm 드리프트 — 한 시간에 수십 ms 누적).
2. **정밀 동기화**: 양쪽에서 추적한 **공 궤적(필드 좌표)의 시간 정렬**
   — 음속 무관, 상관 피크 날카로움. 기존 분석 산출물끼리 맞추는 것.

### 공간 정합 — 소스별로 다름

- **고정 리그(파노라마)끼리**: 필드 좌표계가 공통 언어 — 직사각형
  대칭 4가지(항등/X반전/Y반전/180°) 중 판별만 (공 궤적 몇 초 or
  카메라 사이드 사용자 입력).
- **AX700**: **팬 위주, 줌 드묾 (사용자 확인)** → 고정 위치 회전
  카메라 모델. 프레임 간 관계가 H = K′·R·K⁻¹ 이라 특징점 기반
  호모그래피 분해로 **회전 R 과 초점거리 비율 f′/f 를 동시 추정**
  (회전 카메라 자가 캘리브레이션) — 줌도 예외가 아니라 추적
  파라미터 하나. 옵티컬 플로우 발산은 싼 줌 감지 신호로 병용.
  SfM 전체는 부적합 (삼각대 = 병진 0 → 시차 퇴화) — 회전+줌
  호모그래피가 정확히 맞는 모델.

### 데이터 모델 — match 컨테이너 (신규 레이어)

영상 중심(사이드카) 모델은 유지하고 위에 얹는다:
`<경기>.match.json` = 멤버 영상 목록 + 시계 모델(a, b) + 공간 정합 +
융합 트랙. 비파괴 원칙 그대로 — 각 영상의 분석/에디트는 불변.

## 단계

### 1. 동기화 코어

- 호각 상호상관 + 드리프트 선형 모델 (`core/sync_multi.py`).
- AX700 오디오에서 whistle.py 실행 가능 확인 (마이크 특성).
- 합성 테스트: 알려진 오프셋+드리프트 주입 → 복원 오차 < 1 프레임.

### 2. AX700 파악 + 분석 패스

- 실제 영상 특성 조사: 팬 위주·줌 드묾은 확인됨 — 남은 항목: 오디오
  유무/게인, 시간대 커버리지, 삼각대 고정 여부(위치 불변 가정).
- analyze_video 를 AX700 에 적용 (파노라마 가정 — field_top_frac,
  해상도 — 점검). 사이드카 체계 동일 적용.
- **조기 성과물**: 동기화만 되면 융합 전에도 — 하이라이트 일괄
  내보내기에 "AX700 앵글 클립 동시 추출" 옵션 (고화질 대체 앵글).

### 3. AX700 필드 정합 (회전+줌 추적)

- 기준 프레임 몇 장에 사용자 랜드마크(기존 UI 재사용) → 내부
  파라미터(f₀)·설치 위치·기준 회전 추정.
- 이후 프레임은 **기존 스티칭 정합 스택 재사용**: SIFT 매칭 + RANSAC
  (자동 정합에서 잔차 ~0.13° 검증됨)으로 프레임 간 H 추정 →
  H = K′RK⁻¹ 분해로 회전 3자유도 + f 비율 동시 추적. 옵티컬 플로우
  발산을 줌 감지 보조 신호로.
- 누적 드리프트는 흰 라인/랜드마크 재정렬로 주기 보정.
- 정합 신뢰도 낮은 구간(급팬 블러·가림)만 품질 플래그로 융합 제외.

### 4. 공 융합 + 3D 삼각측량

- 근접 매칭 → 카메라별 거리 가중 융합. 원경 공백 상호 보완.
- 두 시점 삼각측량으로 공중볼 3D — 단일 카메라 탄도 피팅의
  z<0.85h 한계 제거 (airborne.py 대체 경로).

### 5. 선수/심판 매칭 — 통합 ID

- 샘플별 헝가리안: 필드 위치 + 유니폼 색 + **등번호** (양쪽 OCR 로
  전 선수 커버 — 근측 게이팅이 반대편에서 완성됨).
- 교차 시점 검증으로 트랙릿 병합 품질 향상 (오병합 탐지).
- 통합 ID 는 match 레이어에만 — 영상별 tid 불변.

### 6. PitchWatch UI — 경기 열기

- match.json 열기: 통합 타임라인/레이더 (융합 트랙), 미리보기 카메라
  전환, 이벤트/하이라이트는 경기 레벨로 승격.

### 7. 파생 지표 (융합 품질 확인 후)

- 점유율(공-선수 근접 지속 추론), 패스맵, 오프사이드 라인 검증,
  전 구간 균일 정확도의 이동거리/히트맵.

## 완료 기준

- 1: 합성 오프셋+드리프트 복원 < 1프레임, 실경기 호각↔공궤적 정렬 일치.
- 2~3: AX700 구간에서 공/선수 필드 좌표가 파노라마와 rms 수 m 내 일치.
- 4: 파노라마 원경 공 커버리지(검출률) 개선 정량 확인.
- 5: 통합 ID 로 22+3명 로스터 대응, 오병합 교차 탐지 사례 확인.
- 조기 성과: 하이라이트 클립의 AX700 고화질 앵글 추출 동작.

## 확장 시나리오 — 카메라 대수 (타 세션 논의 흡수)

같은 주제가 다른 세션에서 "8대 리그" 논의로 나왔음 — 결론을 여기로
합류시킨다 (별도 계획 문서 만들지 말 것, P05 는 앱 분리로 사용됨):

- **현재 (2대 리그 + AX700)**: P06 본문 — AX700 이 반대편 시점 융합의
  실데이터 검증대. 비용 0, 팬 카메라라 상시 전역 커버는 아님.
- **4대 (맞은편 리그 추가) = 스위트스팟**: 근/원측은 폭 방향 분할이라
  맞은편 리그 하나로 모든 지점이 어느 한 리그의 근측이 됨 — OCR·
  ReID 전 구역화, 가림 해소, 상호 원측 보완. 리그↔리그 정합은 대칭
  판별뿐이라 AX700 케이스보다 쉬움 (P06-1 동기화는 그대로 적용).
  AX700 융합 결과가 좋으면 이것이 구매 근거.
- **8대 (골문 뒤 추가)**: 골에어리어 디테일·삼각측량 정밀도. 단
  트래킹/OCR 한계 해소는 4대에서 대부분 완료. 비용은 선형 이상 —
  촬영 준비 4배, 1대 고장 = 구멍 (20250420 R캠 사례), ~130GB/시간,
  처리 리그당 ~4시간, association 복잡도 증가. **4대 융합이 안정된
  뒤 판단.**
- **대안**: 원측 해상도가 목적이면 대수 대신 카메라 급 (HERO5 4K →
  5.3K = 선형 ~1.7배). OCR 문턱(90px)에는 부족하나 추적 품질 향상,
  운영 복잡도 불변.

## 위험/미결

- 줌은 f 동시 추적으로 흡수 (호모그래피 분해) — 남는 예외는 삼각대
  이동(위치 불변 깨짐)뿐이며 세그먼트 재캘리브레이션 폴백.
- 오디오 게인/AGC 차이로 호각 상관이 약할 수 있음 — tonal 트랙 사용.
- 장기: 3카메라 이상 일반화는 이 구조(공통 필드 좌표 + match 레이어)
  그대로 확장 가능.
