# 006: 원근 보정 원패스 렌더링 (스티칭 remap 합성) + GUI 노출

- 날짜: 2026-07-18
- 종류: 작업 기록
- 관련: [005](20260718_005_verification_perspective.md)

## 동기

005 의 `pano_perspective.py` 는 완성 파노라마를 다시 리샘플+재인코딩하므로
세대 손실이 쌓인다. 원근 맵을 원통 스티칭 remap 과 합성하면 소스에서
보간 1회·인코딩 1회로 보정된 파노라마가 나온다 (이론상 최고 화질).

## 구현 (`core/render.py`)

`Renderer(persp_k=, persp_m=)`: 원통 맵 생성 직후 원근 맵으로 리샘플
(`cv2.remap` 을 맵 자체에 적용 — refine_seam 의 국소 보정과 같은 기법).
합성을 __init__ 최상류에 두면 마스크·심 가중치·게인·refine_seam 이
전부 합성 이후의 출력 공간에서 동작해 하류 수정이 거의 없다.

성립 근거:
- **심이 중앙 열**이라 키스톤(x 를 중심 기준 행별 스케일) 후에도 수직 유지.
- 수평선은 elevation=0 행 `(out_h-1)·tan(el1)/(tan(el1)-tan(el0))` 로 정확.
- 심 밴드는 상단에서 소스가 가로로 늘어나는 만큼 `m` 배 확장.
- 페더 폭이 출력 공간에서 균일해짐 (기존: 파노라마 공간 균일) — 무해.

GUI: 내보내기 탭에 체크박스 + k/m 스핀박스. 미리보기에도 반영되고
프로젝트 JSON(`export.perspective`)에 저장/복원.

## 검증

- 유닛 테스트: 원패스 == "일반 렌더 → PerspectiveWarp 투패스" (동일
  이미지·동일 회전으로 심 기여 제거, 렌더 스케일 기준 충분히 블러한
  합성 패턴). 전체 19개 통과.
- 테스트 설계 교훈 2가지:
  - 두 카메라에 서로 다른 랜덤 이미지를 주면 심 페더 폭 차이(의도된
    동작 차)가 diff 를 지배해 합성 정확도를 못 본다.
  - scale=0.2 렌더는 5배 다운샘플이라 소스에 고주파가 남으면 앨리어싱
    때문에 보간 1회/2회가 크게 벌어진다 (균일 diff ~7). σ=8 블러로 해결.
    거꾸로 말하면 실제 영상에서도 원패스가 그만큼 선명하다는 뜻.
- GUI 배선은 offscreen 인스턴스화로 스모크 테스트 (저장/복원 왕복 포함).
