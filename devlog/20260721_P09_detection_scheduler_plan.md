# P09: 검출 통합 스케줄러 — 디코드 1회, 소비 N개

- 날짜: 2026-07-21
- 종류: 계획 (사용자 질문: 여러 detection 을 가장 효율적으로)
- 관련: [P08](20260721_P08_match_metrics_plan.md) (포즈 확장),
  [039](20260719_039_role_latency_profile.md) (프로파일링 이력)

## 진단

검출 종류가 늘었다: 전역 YOLO(공/선수) + 갭필 재검출 + 등번호 OCR +
(예정) 포즈. 현 구조는 **패스마다 파노라마를 재디코드** — 5906px
H.264 디코드가 공통 병목이고 GPU 사용률 57% 가 그 증거 (추론이
디코드 대기). 모델 추가 = 패스 추가로 선형 악화.

## 설계

1. **단일 센싱 패스**: 디코드 루프 1개 + 소비자 구독 —
   전역 검출(3프레임마다)이 1차, 그 출력이 크롭 소비자(OCR·포즈)를
   게이트. 갭필은 별도 패스가 아니라 "미해결 구간 재방문 큐".
2. **게이트 + 예산**: OCR = 높이 ≥90px & 트랙 미해결 (per_track 예산
   현행). 포즈 = **이벤트 트리거** — 공 근접 ±2s 창의 근측 선수만
   (킥 판정용 P08-2). 풀경기 포즈는 하지 않는다.
3. **크롭 배치**: 프레임 경계 무시하고 크롭을 모아 GPU 배치 추론
   (OCR·포즈 공통) — 노는 GPU 43% 흡수.
4. **모델 통합은 선별**: 전역 det 유지 (공 필요). yolo11-pose 로
   사람 det 대체는 4대 리그(전 구역 근측) 때 재평가.
5. **디코드 가속**: NVDEC(Windows)/프리페치 스레드로 디코드-추론
   중첩. 프록시(1920px) 재사용 검토 — OCR·포즈는 원본 크롭 필수,
   전역 det 는 이미 det_w 축소라 원본 디코드가 유일한 이유.

## 단계

1. analyze_video 에 크롭 소비자 훅 (콜백 목록) — OCR 을 첫 이주
   대상으로 (별도 패스 제거, 헤드리스 시간 실측 비교).
2. 포즈 소비자 (P08-2 와 함께) — 이벤트 트리거 창.
3. 갭필 재방문 큐 통합.
4. 디코드 프리페치/NVDEC (Windows 실측 후).

완료 기준: 헤드리스 경기당 처리 시간 (현재 분석+OCR 분리 대비)
단축 실측 + 산출물 동일성 (사이드카 diff).
