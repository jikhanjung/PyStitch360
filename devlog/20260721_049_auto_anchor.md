# 049: 필드 마킹 자동 앵커 — 세 번 실패하고 임무를 다시 정의했다

- 날짜: 2026-07-21
- 종류: 구현/알고리즘
- 관련: [P06](20260720_P06_multicam_fusion_plan.md) 3단계 자동화 방향,
  [048](20260721_048_rotcam_realfootage.md)

## 목표

체인 추적의 누적 드리프트를 수동 랜드마크 없이 제거: 근사 자세로
필드 템플릿을 투영 → 흰 선 중심 탐색 → (R, f) 재정렬 (rotcam.auto_anchor).

## 실패의 기록 (합성 검증이 전부 잡아냄)

1. **점 대 점 ICP (Kabsch)**: 정체. 선 위 대응은 접선 변위에 "오차
   0" 을 주장해 보정을 되끈다 (aperture) — 회당 10~15% 만 수렴,
   내부 반복은 오히려 틀린 고정점에 잠김 (f-회전 결합 국소해).
2. **직선 인시던스 DLT** (l^T G x = 0, 8-DOF 일발): 수학은 완전
   데이터에서 정확(오차 0.0) — 그러나 (a) 원측(카메라 ~79m)은 역투영
   증폭으로 필드 스냅 라벨이 23% 오대응, (b) 라벨을 탐색 시드로
   바꿔도 수평선 부근 라인 밀집(터치라인↔박스 ~21px)이 큰 탐색
   창(48px)의 중심 검출 자체를 오염, (c) 원측을 잘라내면 직선
   패밀리가 3개뿐이라 8-DOF 가 미정 — **미지수는 R+f=4 인데 8 을
   풀고 있었다**.
3. **4-파라미터 point-to-line 가우스-뉴턴** (cam_pos 고정): 완전
   데이터에선 정확하나 실검출에선 **f 폭주** (1400→1766) — 제한된
   시야에서 (f, R) 준퇴화 방향이 있어 검출 노이즈만으로 잔차가 더
   낮은 가짜 해에 도달한다.

## 최종 설계

- **임무 재정의가 핵심 수정**: auto_anchor 는 큰 섭동의 전역 복원이
  아니라 **주기 드리프트 보정** (앵커 간 ≤0.5°) — 전역 복원은 단일
  뷰 라인 기하로는 원래 불량조건이고 기준 캘리브레이션의 몫이다.
- 4-파라미터 GN + **f 사전 제약** (초기 f σ≈3% — 체인의 줌비 추적은
  043 실측 80스텝 f<2% 로 신뢰 가능): 준퇴화 방향만 고정, 관측
  가능한 방향은 데이터가 지배.
- 탐색 반경 16→10→8px 축소 스케줄 — 작아서 라인 밀집에서 이웃을
  줍지 않고, 임무 규모(0.5°≈12px)에는 충분. 라벨 = 탐색 시드
  샘플의 패밀리. 수평선 부근(지면 거리 80m+)만 제외.
- 잔차 = 역투영점의 소속 직선 부호 거리 / 서클 반경 편차,
  1/지면거리 가중(≈각도 오차 등화) + Huber.

## 검증 (tests/test_rotcam.py — 121개 전체 통과)

- 현실 규모 섭동(팬 0.35°+틸트 0.2°+f 1%) → 회전 오차 <0.1°,
  f <1%, 지면 재투영 <4px 복원.
- 라인 없는 화면(관중석 팬) → None (품질 플래그 경로).

## 교훈

- 격리 테스트(완전 데이터)로 "수학 대 데이터" 를 즉시 분리할 것 —
  세 접근 모두 수학은 맞았고 문제는 항상 대응/조건이었다.
- 풀리지 않으면 미지수 수부터 세라 — 8-DOF DLT 는 4-DOF 문제의
  과잉 일반화였다.
- 알고리즘이 임무보다 야심차면 임무를 다시 봐라 — "드리프트 보정"
  에는 작은 탐색 반경이 정답이었고, 큰 반경은 오염만 불렀다.
