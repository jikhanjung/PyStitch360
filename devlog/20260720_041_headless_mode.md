# 041: 헤드리스 모드 — 디렉터리 2개로 스티칭→분석→OCR 무인 실행

- 날짜: 2026-07-20
- 종류: 기능
- 관련: [040](20260719_040_jersey_ocr.md) 등번호 OCR

## 사용법

```
python main.py --headless <left_dir> <right_dir> [--out DIR] [옵션]
```

좌/우 GoPro 디렉터리만 주면 짝 맞춤 → 오디오 동기화 → 자동 정합(+
드리프트 감시) → 파노라마 인코딩 → 선수/공 분석 → 등번호 OCR 까지
무인 실행. 출력 디렉터리는 기본으로 **두 디렉터리 이름의 공통 토큰**
(예: `20260712_GoPro5_L`/`_R` → `<부모>/20260712_GoPro5`)에 만들고,
산출물: `pano_XXXX.mp4` + `.pystitch.json`(GUI 로 열기 가능) +
`.analysis.json` + `.events.json`("ocr_numbers" 제안).

## 좌/우 짝 맞추기 (core/pairing.py)

- 각 디렉터리를 `group_directory` 로 챕터 체인 단위 그룹핑
  (GOPRxxxx → GP01xxxx → …).
- 동시 녹화라 총 길이가 거의 같음 → **챕터 체인 총 파일 크기의
  상대차**를 비용으로 그리디 매칭. 챕터 수 차이는 0.05/개 가산.
- 비용 0.25 초과 조합은 짝 없음으로 보고만 하고 건너뜀.

## 자동 정합 + 드리프트 감시 (headless.py)

- **설치 완료(settle) 탐지**: 녹화 시작 후 각도 조절·삼각대 세우는 몇
  분(사용자 확인)이 자이로 방향 변경 이벤트로 나타난다 — 초반
  15분(_SETTLE_WINDOW) 안의 마지막 persistent 이벤트 + 30초 이후에서
  동기화·정합. 스모크 실증: t=3s 정합 → 18s/38s 리그 조작으로 중앙선이
  왼쪽·기울어짐 (t=5s 프레임은 정확히 센터 — 하프라인 검출 자체는 정상).
- 오디오 상관 동기화: settle+60s 우선, 신뢰도 <4 면 다른 구간 재시도,
  최고 신뢰도 채택.
- 초기 정합은 settle 이후 구간 **앞쪽 우선**(5%→20%→…) — 스캔이
  앞으로만 진행하므로.
- **5분(--check-every) 간격 드리프트 체크**: 겹침 영역 SIFT 매칭
  광선쌍에 현재 상대회전(Rh²)을 적용한 각오차의 중앙값. 문턱 =
  max(--drift-thresh 0.5°, 초기 정합 기준 잔차×3).
- 틀어졌으면 (정상, 틀어짐) 사이 **이진 탐색**(10초 해상도)으로 경계
  탐지. **자이로(GPMF) 스파이크 중 persistent(방향 변경) 이벤트**가
  구간 안에 있으면 그 직전/직후 잔차로 먼저 확인해 탐색 생략.
- 경계 직후(+5/35/65/125s) 프레임에서 `reuse_level=초기 정합` 으로
  재정합 → 새 세그먼트. 수평/센터링 재사용이라 세그먼트 간 뷰·출력
  크기 유지 (GUI 세그먼트 규약 그대로).

## ExportWorker 코어 분리 (core/export.py)

GUI `ExportWorker._run` 의 3단 파이프라인(읽기→렌더→인코더 쓰기)을
`export_pano()` 로 추출 — Qt 의존 없음, 진행/로그/취소는 콜백.
ExportWorker 는 이제 이 함수를 시그널로 감싸기만 한다.

동작 변경 하나: **게인·심(seam-refine) 보정 측정을 내보내기 시작/경계
프레임이 아니라 각 세그먼트의 정합 프레임(align_sec)에서** 한다.
스모크 테스트에서 t=0 프레임(무늬 없는 잔디만)으로 측정하다 seam-refine
이 "측정 타일 부족" 으로 생략되어 심 세로 어긋남이 그대로 노출됐다 —
정합이 성공한 프레임이면 특징이 충분하다. 보정 프레임들은 스트리밍
시작 전에 미리 읽어둔다 (경계 재구성 시 reader 스레드와 디코더 경합
방지). GUI 내보내기도 동일하게 개선된다.

## OCR 폴백 (core/ocr.py)

`collect_ocr_candidates(calib=None)` 허용: 필드 캘리브레이션이 없는
헤드리스 단계에서는 근측 필드 게이트 없이 **박스 높이 게이트만** 적용
(원경 선수는 min_h=90px 로 대부분 걸러짐). 팀 분류는 `classify_teams`
자동(k-means), 병합/역할 시드 없음. 결과는 기존과 같이 제안으로만
저장 — GUI 번호 메뉴 "OCR 제안" 으로 적용.

## 재실행 안전

pano.mp4 / .analysis.json / events "ocr_numbers" 가 이미 있으면 해당
단계 건너뜀 (--force 로 강제 재실행). 분석은 기존 체크포인트
(.analysis.part.json) 재개 그대로. 한 쌍이 실패해도 다음 쌍 계속.

## --auto-el: 세로 범위 자동 결정

"촬영된 화면 다 나오게 상하 화각을 넓히면?" 논의 결과: 원통 투영은
세로가 tan(el)·f 라 전체 커버(−70..+45°)는 픽셀 3.2배 + 극단 고도
스트레칭이라 기각. 대신 커버리지 실측: 두 카메라 마스크 union 에서
상/하 각각 **열의 10%까지 코너 블랙 허용**하는 최대 행 구간을 el 로
환산. 전-열 커버(블랙 0%)를 요구하면 파노라마 좌우 극단 열 때문에
−36..−12° 로 무의미하게 좁아진다. 10% 허용 실측 −48.2..+4.7° 는
기본값(−45..+10)과 거의 일치 — 기본값이 이 리그에 근사 최적임을 교차
확인. 기본은 여전히 고정 범위, `--auto-el` 은 옵트인.

auto-level 스텝 한계 ±25°→±35° (경고 문턱 24→34°): 삼각대+마스트
리그의 실측 pitch +28.9° 가 정상 값인데 경고가 떴었다.

## 남은 것

- 드리프트 문턱(0.5°)·이진 탐색 해상도(10s)는 실제 범프 사례로 보정
  필요 — 아직 실측 범프 데이터 없음.
- 자이로는 좌측 카메라만 읽음 — 우측 단독 범프는 이진 탐색으로만.
