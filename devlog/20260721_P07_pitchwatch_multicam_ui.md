# P07: PitchWatch 멀티캠 인터페이스 계획

- 날짜: 2026-07-21
- 종류: 계획
- 관련: [P06](20260720_P06_multicam_fusion_plan.md) (백엔드 로드맵),
  [P05](20260720_P05_app_split_plan.md) (PitchWatch 소속),
  [046](20260721_046_p06_sync_realdata.md) (실경기 동기화 성립 — 이
  계획의 착수 조건)

## 배경/원칙

- 20241020 실경기에서 파노라마↔AX700 거친 동기화가 전·후반 모두
  성립 (±0.2s) — "동기화된 멀티캠"이 이제 실데이터다.
- PtzTab 은 단일 영상 + 사이드카 중심의 단일 클래스(~4천 줄)로, 영상
  상태·편집 상태·워커가 결합돼 있다. **이걸 N-카메라로 일반화하는
  리팩토링은 하지 않는다** — 편집(검수·키프레임·역할)은 지금처럼
  주 파노라마 하나에만 있고, 다른 카메라는 **읽기 전용 뷰어**로
  얹는다. 융합(P06-4~5)이 안정되기 전에 편집 표면을 넓히면 비파괴
  원칙(영상별 사이드카 불변)과 충돌한다.
- 백엔드가 익는 순서(P06: 동기화→정합→융합)대로 UI 도 3단계로
  나눠, 각 단계가 그 시점에 실제로 동작하는 기능만 노출한다.

## 데이터: `<경기>.match.json` v1 (신규, P06 설계의 부분집합)

```json
{
  "version": 1,
  "title": "20241020 준결승",
  "halves": [
    {"label": "전반", "primary": "pano_5316.mp4",
     "alts": [{"video": ".../C0011.MP4",
               "clock": {"offset": 1872.26, "drift": 1.000187}}]},
    {"label": "후반", "primary": "pano_5317.mp4",
     "alts": [{"video": ".../C0012.MP4",
               "clock": {"offset": 191.04, "drift": 1.0000676}}]}
  ]
}
```

- half 단위 구성: 현행 촬영 관행(하프별 파일)과 per-video sync
  (.events.json "sync")를 그대로 담는 최소 형태. 시계 모델은 sync
  결과의 사본 (match.json 만 있으면 열림).
- 공간 정합(rotcam)·융합 트랙은 v2 에서 추가 — 스키마에 자리만
  예약하고 v1 은 만들지 않는다 (P06-3~5 전에 정의해봐야 추측).
- 저장 위치: primary 옆 (`20241020_GoPro5_Match1.match.json` 등).

## 1단계 — 동기화 뷰어 (지금 착수 가능)

**1-1. 경기 열기/만들기**

- PitchWatchWindow 파일 메뉴: "경기 열기(.match.json)...", "최근 경기"
  (최근 파노라마 목록과 병렬), "멀티캠 경기 만들기...".
- 만들기 대화: primary 파노라마들(하프 순서) + alt 영상들을 추가하면
  각 쌍의 `.whistle.json`/`.events.json "sync"` 를 찾아 시계 모델을
  채우고, 없으면 그 자리에서 whistle→sync_cams 실행 (워커, 로그 탭).
  버튼 하나로 "이 경기의 AX700 붙이기"가 되는 게 목표.

**1-2. 동시 표시 뷰어 — PiP + 좌우 분할 + 전환 (사용자 확정, 2026-07-21)**

- 보기 모드 3가지, 페인 우상단 오버레이/단축키로 전환:
  - **PiP (기본)**: alt 를 파노라마 페인 **상단 구석** 오버레이로 —
    초광폭 파노라마의 상단 모서리는 관중석/하늘이라 가림 손실이
    작다. 드래그 이동·모서리 리사이즈, 위치/크기는 QSettings.
  - **좌우 분할**: QSplitter 가로 배치 — 파노라마 | alt. 와이드
    창(1920px)에서 파노라마 ~1150×380 + AX700 ~770×430 으로 둘 다
    판독 가능.
  - **전환**: 메인 페인을 alt 로 통째 교체 (`1`=파노라마, `2..9`=alt
    단축키) — alt 를 크게 봐야 할 때 (등번호·디테일 확인).
- 어느 모드든 시간축·타임라인·단축키·편집 입력은 전부 **primary
  페인 기준** 그대로. alt 페인은 읽기 전용 FramePane (클릭 무시).
- 구현: alt 별 `cv2.VideoCapture` 지연 생성 + `to_other_time` 환산
  시크. 갱신 정책 — 정지/시크 시 즉시, **재생 중엔 저주기(~5fps)**
  로 따라가기 (이중 풀레이트 디코드 회피). 4K H.264 랜덤 시크가
  무거우면 기존 ProxyWorker 재사용 ("alt 프록시 생성" 메뉴).
- alt 오디오 재생 전환은 v1 범위 밖 (위험/미결 참조).
- **확장 대비 (사용자 요구, 2026-07-21)**: 카메라가 늘면(4~8대,
  P06 확장 시나리오) 화면 구성이 더 자유로워야 한다 — 구현을
  "alt 뷰 목록 + 배치 전략(모드)" 구조로: 모드는 alt 목록 위에서
  동작하는 레이아웃일 뿐이라, v2 에서 새 모드로 추가할 수 있다.
  v1 의 PiP/분할은 "선택된 alt 하나" 를 보여주는 최소 전략.
  사용자가 든 v2 레이아웃 예시:
  - 상하 2분할 — 위 파노라마, 아래를 다시 3분할해 개별 카메라 3대
  - 상하 2분할 — 반대편 사이드라인의 파노라마 두 대 (4대 리그 시나리오)
  - 4분할 쿼드뷰 — 카메라 4대 동시
  이를 위해 디코드 워커는 처음부터 **카메라별 요청 슬롯** (다중 페인이
  각자 request — 워커 수정 없이 그리드 가능), alt 페인은 목록으로
  관리한다. v2 는 "레이아웃 = 슬롯 배열 + 슬롯별 카메라 할당" 정도의
  선언적 정의(프리셋 몇 개 + 커스텀)로 가는 게 자연스럽다.

**1-3. 타임라인 커버리지 레인**

- TimelineView 에 카메라 레인 1줄 추가: alt 영상이 primary 시간축
  어디를 덮는지 스팬 표시 (+호각 매칭 쌍 눈금). 클릭 = 그 시각으로
  시크 + 해당 카메라로 전환.
- 하프 사이 갭/비커버 구간이 한눈에 보인다 (20241020 사례: C0012 가
  pano_5317 보다 5분 먼저 시작 → 앞쪽 5분은 AX700 만 있음... 역방향
  커버리지는 v1 표시 대상 아님, primary 기준만).

**1-4. 하이라이트 대체 앵글 (기존 기능 노출 확대)**

- 하이라이트 목록 항목에 앵글 뱃지 (동기화된 alt 가 그 구간을 덮으면
  "AX700" 표시). 일괄 내보내기의 기존 체크박스는 그대로.

완료 기준: 20241020 경기를 match.json 으로 열고, 임의 시각에서 PiP/
좌우 분할/전환 3모드로 AX700 동일 장면 확인(±0.5s), 재생 중 PiP
저주기 추종, 커버리지 레인 클릭 시크, 하이라이트 내보내기에서 alt
클립 동반 생성.

## 2단계 — 필드 정합 후 (P06-3 완료가 전제)

- alt 카메라 레이더: rotcam 캘리브레이션(.rotcam.json)이 있으면 alt
  표시 중에도 미니맵/필드 좌표 오버레이 유지.
- 정합 품질 플래그 구간(급팬·블러)을 커버리지 레인에 음영 표시.
- alt 뷰에 공/선수 박스 오버레이 (alt 자체 분석 사이드카가 있을 때).

## 3단계 — 융합 후 (P06-4~5 완료가 전제)

- 융합 레이더: 미니맵을 match 레이어 융합 트랙으로 전환 (원경 공백
  메워진 궤적). 카메라별/융합 토글.
- 통합 ID: 선수 패널이 match 레이어 통합 ID 기준으로 표시, 영상별
  tid 는 상세에서만.
- 이벤트/하이라이트의 경기 레벨 승격 + PTZ 내보내기에서 구간별 최적
  앵글 자동 선택(공 위치 기준 근측 카메라).

## 구현 노트

- MatchContext 객체 (신규, gui 또는 core): match.json 로드/저장,
  멤버·시계 모델 보관, `frame_of(cam, primary_frame)` 환산. PtzTab 은
  `open_match(ctx)` 로 받고, 내부는 기존 pano 경로 그대로 (primary 를
  open_path). 하프 전환은 v1 에선 "경기 메뉴 > 전반/후반" 으로
  primary 를 갈아끼우는 것 (PtzTab 상태 재사용, 기존 open_path 경로).
- PitchWatch 전용 기능이므로 pitchwatch.py / PtzTab 에만 — 통합
  main.py 에서도 동작하지만 PitchStitch 는 무관.
- 1단계는 P06-3(rotcam 실영상)과 독립 — 병행 가능.

## 위험/미결

- 46GB pano + 27GB 4K alt 의 동시 열림: 메모리는 캡 2개라 문제없고
  시크 지연이 관건 — 프록시로 해소 (기존 검증된 경로).
- 하프 경계 시각의 자동 인식(경기 시계 vs 파일 시계)은 match_info
  앵커 재사용 — v1 에선 하프 = 파일 단위라 필요 없음.
- AX700 오디오 전환(재생 소리를 alt 소리로)은 v1 범위 밖.
