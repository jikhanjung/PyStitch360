## 2. 내보내기 성능

| 구성 | 속도 (WSL CPU, 5906px) |
|---|---|
| 순차 루프 (기준) | 6.8 fps |
| + 읽기/렌더/쓰기 파이프라인 | 6.0 fps (개선 없음 — CPU 총량 병목) |
| + **분할 렌더링** | **8.25 fps** |

교훈: 지연 중첩(파이프라이닝)은 디코딩·remap·x264 가 같은 코어를 두고 경쟁하는
CPU-bound 상황에서 효과 없음. 총 작업량을 줄여야 함 → 심 좌/우는 단일 카메라만
쓰는 성질을 이용해 **각 카메라를 자기 절반 + 심 밴드만 워핑** (render 75→30ms).
파이프라인 자체는 유지 (NVENC 등 인코딩이 가벼워지면 효과 발생).

NVENC: `ffmpeg -encoders` 를 검사해 가용한 경우에만 콤보에 노출
(`core/encoders.py`). NVENC 는 CRF 대신 `-rc vbr -cq`. WSL 에는 없고
Windows 네이티브에서 확인 필요.

## 3. 세그먼트 (충격에 의한 방향 변동)

- 세그먼트 = (시작 시각, Alignment). 첫 정합은 0초부터, 이후 정합은 현재
  프레임부터 적용. 같은 지점(±0.5s) 재정합은 교체.
- 정합 탭에 목록 + 이동/삭제. 미리보기는 현재 시각 담당 세그먼트 사용.
- 내보내기: 경계 프레임에서 렌더러 재구성 (그 시점 프레임으로 게인·심 보정
  재추정). 출력 크기 고정을 위해 yaw 범위 폭은 첫 세그먼트 기준.
- 검증: 300~306초 내보내기에 303초 경계 → 정상 전환.
- 자동 감지(GPMF 자이로/앵커 추적)는 추후 — 현재는 수동 재정합 워크플로우.

## 4. 가상 PTZ (`core/ptz.py`)

PitchAnalysis 의 아이디어 재구현: YOLOv8n(공 클래스 가중 30배) → 가중평균
목표점 → EMA(α=0.08) + 프레임당 최대 이동(25px) 스무딩 → 1920×1080 크롭.
감지는 5프레임마다 1280px 축소본에서 수행.

- ultralytics 는 **선택 의존성** (미설치 시 기능만 비활성). 가중치는
  `presets/yolov8n.pt` (PitchStitch 에서 복사).
- 15초 내보내기 테스트: 8.7fps, 크롭 창이 경기 흐름 추종 확인.
- 파노라마 50% 해상도에서는 1080p 크롭 불가(세로 부족) → 100% 필요, 에러 안내.
